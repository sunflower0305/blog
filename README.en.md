# Leyang Blog

If you want a place for learning, writing, and sharing that truly belongs to you, instead of relying entirely on platform algorithms, this project is built for exactly that.

Leyang Blog is not just a static Markdown theme. It is a complete blog system with dual editors for frontend and admin, AI writing assistance, AI image generation, theming, full-text search, API tokens, and an external publishing ecosystem, all wired up so it is easier to keep writing consistently.

- Live demo: <https://blog.zhangleyang.com/>
- Intro article: <https://blog.zhangleyang.com/2026-07-10-uxI_bn>
- Project docs: <https://zread.ai/sunflower0305/blog>

## Why Build a Site of Your Own

- Social media accounts can be suspended and platform traffic can change overnight, but your own site stays yours
- A writing system should feel light enough that you can open it and start writing without friction
- AI is most useful for repetitive work like summaries, tags, covers, slugs, and generated images
- A blog should be more than a showcase page. It should become a long-term knowledge asset

## What You Get

- Edit from both the public side and the admin side, with a WYSIWYG experience close to Feishu / Notion
- Three homepage themes, mobile-friendly, ready to use out of the box
- Bubble Menu + Ask AI for rewriting, polishing, expanding, and translating selected text
- AI-assisted summaries, tags, SEO slugs, and cover image generation
- AI image model presets, prompt templates, recent generation history, and insert / replace workflows
- Right-click image actions: download, set as cover, align, crop, and use as image generation reference
- Publication states: public, draft, password protected, and link-only
- Default initial presets for theme, navigation, fonts, AI text model templates, and AI image model templates
- Deployment on Cloudflare Workers + D1 + R2, without maintaining your own server or CDN

## One-Command Deploy to Cloudflare

```bash
pnpm run deploy
```

This template is already prepared for the Deploy Button flow:

- Cloudflare reads the Worker config from the repository
- It automatically provisions the required `D1` / `R2` bindings
- It uses the custom deploy script included in the repo
- It applies the database schema and default template settings during deployment

It is recommended to prepare these values before deployment:

- `NEXT_PUBLIC_SITE_URL`
- `ADMIN_PASSWORD`
- `ADMIN_TOKEN_SALT`
- `AI_CONFIG_ENCRYPTION_SECRET`
- `AI_API_KEY` (optional)

If you prefer managing your Cloudflare resources manually, you can also deploy via CLI:

```bash
pnpm install
cp .env.example .env.local
pnpm exec wrangler login
pnpm run cf:init -- --site-url=https://your-domain.com
pnpm run cf-typegen
pnpm run deploy
```

`pnpm run cf:init` generates or updates `wrangler.local.toml` and prepares the `DB`, `IMAGES`, and `CACHE` bindings required by Vinext.

## Local Development

```bash
git clone https://github.com/sunflower0305/blog.git
cd blog
pnpm install
cp .env.example .env.local
pnpm run dev
```

`pnpm run dev` uses the local Workers runtime by default and does not connect to remote Cloudflare bindings, so it starts faster. If you need to debug Workers AI or real remote bindings, run `pnpm run cf:init -- --site-url=https://your-domain.com` first, then use:

```bash
pnpm run dev:remote
```

### Environment Variable Configuration

[`config/runtime-env.json`](config/runtime-env.json) is the single contract for runtime environment variables. It defines variable names, whether they are sensitive, whether they are used for local preview, and their public defaults. Avoid maintaining separate variable lists in `.env.example` and `wrangler.toml`.

- `.env.local`: actual values for local development and preview, not committed to the repository
- `wrangler.local.toml`: generated by `pnpm run cf:init`, contains Cloudflare resource bindings and local deployment config, not committed to the repository
- Cloudflare Secrets: production-sensitive values such as the admin password and API keys
- `wrangler.toml`: structural config for the Deploy Button and non-sensitive defaults

After changing the environment variable contract, run:

```bash
pnpm run config:generate
pnpm run config:check
```

The first command regenerates `.env.example`, and the second checks whether generated files and `wrangler.toml` stay in sync with the contract. `pnpm run verify` already includes this check.

Common entry points:

- Homepage: `/`
- Admin: `/admin`
- Editor: `/editor`

If you want to preview with the Worker runtime locally:

```bash
pnpm run preview
```

## Default Initial Content

After the first initialization, the template includes these built-in capabilities:

- Default navigation
- Default themes and fonts
- Default categories

| `pnpm run deploy`             | Deploy to Cloudflare Workers                                                  |
